# fb-test-gatekeeper

## Цель
Определи минимально достаточный набор тестов и дай чёткий вердикт `можно/нельзя` для merge/deploy.
Делай именно минимально достаточный прогон, потому что это ускоряет цикл поставки без потери контроля качества.

## Прогрессивная загрузка (не читай всё сразу)
1. Сначала используй этот `SKILL.md`.
2. Для точных команд и окружения загрузи `references/TESTING.md`.
3. Для известных flaky/особых сценариев загрузи `references/tests.md`.
4. Не загружай `tests.md`, пока не обнаружены падения или нестабильность.

## Матрица выбора тестов
| Что изменилось | Какие тесты запускать |
|---|---|
| Только фронтенд (CSS/JS/HTML) | `unit` для фронта |
| API endpoints | `unit` + `e2e` по затронутым API |
| Бизнес-логика/сервисы | `unit` по затронутым модулям |
| Схема БД/миграции | `unit` + `e2e` + `docker:test` |
| Dockerfile/docker-compose | `docker:test` |
| CI-конфигурация | `ci` |
| Непонятно или много изменений | полный прогон: `unit` + `e2e` + `docker:test` |

## Алгоритм работы

### Шаг 1. Классифицируй изменения
Определи тип изменений по diff, потому что без классификации нельзя выбрать минимальный набор тестов.

Пример:
```bash
git diff --name-only main...HEAD
```

Ожидаемый результат:
```text
Список файлов, разбитый по категориям (frontend/api/services/db/docker/ci)
```

Если нет уверенности в классе изменений, выбери более строгий набор (эскалация вверх по матрице).

### Шаг 2. Выбери набор тестов по матрице
Сформируй список команд, потому что прозрачный план прогона снижает риск пропустить критичный слой.

Пример плана:
```text
Changes: API + migrations
Run: bun run test:unit && bun run test:e2e && bun run docker:test
```

### Шаг 3. Запусти тесты
Выполни команды в фиксированном порядке, потому что ранние быстрые сигналы экономят время.

Примеры:
```bash
bun run test:unit
# Ожидаемый результат: all passed
```

```bash
bun run test:e2e
# Ожидаемый результат: all passed or list of failed scenarios
```

```bash
bun run docker:test
# Ожидаемый результат: image build ok + tests passed
```

```bash
bun run ci
# Ожидаемый результат: локальная эмуляция CI без критических падений
```

### Шаг 4. Интерпретируй падения
Раздели реальные регрессии и flaky, потому что решение `можно ли мержить` зависит от природы сбоя.

Действия:
1. Если тест упал в затронутом модуле -> считай регрессией.
2. Если тест известен как flaky -> загрузи `references/tests.md` и проверь признаки flaky.
3. При реальной регрессии помоги починить, затем повтори минимально необходимый прогон.

Пример:
```text
Fail in src/api/orders.ts changed in this branch -> regression, block merge/deploy
```

### Шаг 5. Выдай вердикт
Сформулируй однозначный итог, потому что пользователю нужен операционный ответ.

Формат вердикта:
```text
Тесты пройдены. Можно мержить/деплоить.
```

или

```text
Тесты провалены. Нельзя мержить/деплоить.
Нужно исправить:
1) ...
2) ...
```

## Идемпотентность
1. Повторный запуск того же набора тестов не должен менять состояние репозитория.
2. Не расширяй набор тестов без причины между повторными прогонами.
3. Все команды запускай из одного и того же контекста окружения.

## Взаимодействие с другими скилами
1. Если обнаружено, что пользователь переходит к релизу после зелёных тестов, передай управление скилу `fb-deploy-operator` — он лучше справится с этапами деплоя.
2. Если обнаружено, что изменения затрагивают схему БД и нужна миграция, передай управление скилу `fb-migrations-maintainer` — он лучше справится с изменениями схемы.
3. Если обнаружено, что падения связаны с admin API 403/CSRF, передай управление скилу `fb-admin-csrf-guard` — он лучше справится с CSRF-аудитом.

## Когда НЕ использовать этот скил
1. Написание новых тестов с нуля: это задача разработки.
2. Нагрузочное/перформанс-тестирование: отдельный процесс.
3. Чисто деплойный вопрос без тестовой валидации: используй `fb-deploy-operator`.
4. Чисто миграционный вопрос: используй `fb-migrations-maintainer`.

## Decision Tree: выбор минимально достаточного набора
```text
START
  |
  |-- Есть diff по файлам?
  |     |-- Нет -> запроси scope или запусти полный прогон
  |     |-- Да -> классифицируй изменения
  |
  |-- Категория: frontend only?
  |     |-- Да -> unit(front)
  |     |-- Нет -> дальше
  |
  |-- Категория: API/services?
  |     |-- Да -> unit + e2e (затронутые маршруты)
  |     |-- Нет -> дальше
  |
  |-- Категория: DB schema/migrations?
  |     |-- Да -> unit + e2e + docker:test
  |     |-- Нет -> дальше
  |
  |-- Категория: Docker/CI?
  |     |-- Docker -> docker:test
  |     |-- CI -> ci
  |
  |-- Категория неоднозначна?
        |-- Да -> полный прогон
        |-- Нет -> выполнить выбранный набор
```

### Decision Tree: как действовать при падениях
1. Если тест падает в измененном модуле -> блок merge/deploy.
2. Если тест известен как flaky -> сверка с `references/tests.md` и контролируемый rerun.
3. Если падение инфраструктурное (временные network issue) -> rerun + фиксация риска.
4. Если после rerun падение повторяется -> считать как реальную проблему.

Пример:
```text
Fail: e2e checkout flow, затронут src/api/checkout/*
Решение: regression, блок deploy до фикса
```

### Decision Tree: когда эскалировать в другой скил
1. Если решено деплоить после зелёных тестов -> handoff в `fb-deploy-operator`.
2. Если проблема связана с миграциями/схемой -> handoff в `fb-migrations-maintainer`.
3. Если падают admin mutate-flow из-за 403 -> handoff в `fb-admin-csrf-guard`.

## Расширенный протокол запуска тестов
### Фаза A: быстрый сигнал
```bash
bun run test:unit
```

### Фаза B: интеграционный риск
```bash
bun run test:e2e
```

### Фаза C: окружение и контейнеры
```bash
bun run docker:test
```

### Фаза D: CI эквивалент
```bash
bun run ci
```

Пояснение:
1. Если ранняя фаза красная, можно не идти в более дорогие фазы.
2. Для pre-deploy критичных изменений желательно пройти все релевантные фазы.

## Anti-patterns и как исправлять
1. Anti-pattern: всегда запускать полный suite без анализа diff.
   Почему плохо: медленно и дорого.
   Исправление: сначала классификация изменений.

2. Anti-pattern: запускать только unit при изменениях API.
   Почему плохо: пропуск интеграционных регрессий.
   Исправление: добавляй e2e для затронутых endpoint-ов.

3. Anti-pattern: игнорировать docker:test при изменениях Docker.
   Почему плохо: сборка ломается уже в CI/production.
   Исправление: docker:test обязателен.

4. Anti-pattern: считать flaky-тест "нормальным падением".
   Почему плохо: шум и ложные блокировки.
   Исправление: проверяй каталог flaky в `references/tests.md`.

5. Anti-pattern: бесконечные rerun до зеленого.
   Почему плохо: скрытие реальной регрессии.
   Исправление: ограниченный rerun policy.

6. Anti-pattern: нет явного вердикта.
   Почему плохо: пользователь не понимает, можно ли мержить.
   Исправление: формула `pass/fail + причины`.

7. Anti-pattern: запуск тестов в разном окружении без фиксации.
   Почему плохо: нерепродуцируемые результаты.
   Исправление: фиксируй env/команды/ветку.

8. Anti-pattern: запуск e2e без подготовленных зависимостей.
   Почему плохо: ложные падения.
   Исправление: используй `references/TESTING.md` preconditions.

9. Anti-pattern: не связывать падение с diff.
   Почему плохо: неверный приоритет исправлений.
   Исправление: всегда сопоставляй failing area с изменениями.

10. Anti-pattern: после green не делать handoff в deploy.
    Почему плохо: разрыв между quality gate и релизом.
    Исправление: handoff в `fb-deploy-operator`.

## Шаблоны отчетов
### Шаблон: test plan
```text
[TEST PLAN]
date_utc:
branch:
diff_scope:
selected_matrix_row:
commands:
1)
2)
3)
```

### Шаблон: test execution report
```text
[TEST EXECUTION]
unit: pass/fail/not-run
e2e: pass/fail/not-run
docker_test: pass/fail/not-run
ci: pass/fail/not-run
total_duration:
artifacts/logs:
```

### Шаблон: failure triage
```text
[FAILURE TRIAGE]
failing_suite:
failing_test:
module_impacted:
changed_in_branch: yes/no
flaky_signature_match: yes/no
classification: regression/flaky/infra
next_action:
```

### Шаблон: final verdict
```text
[TEST VERDICT]
status: allow-merge-deploy / block
blocking_issues:
1)
2)
confidence_level: high/medium/low
recommended_handoff:
```

### Шаблон: handoff в deploy skill
```text
[HANDOFF]
from_skill: fb-test-gatekeeper
to_skill: fb-deploy-operator
reason: required test set passed
evidence: execution report attached
```

## Stop-conditions
1. Невозможно получить список изменений (`git diff` недоступен).
2. Тестовые команды отсутствуют или не запускаются в окружении.
3. Нет доступа к `references/TESTING.md` при неоднозначных командах.
4. Ветка в broken state и тесты не стартуют технически.

## Критерии готовности к merge/deploy
1. Выбран и обоснован минимальный набор тестов.
2. Выполнены все релевантные команды.
3. Каждое падение классифицировано.
4. Дано бинарное решение `allow/block`.
5. В случае allow указан handoff.

## Готовые команды для быстрой диагностики
```bash
git diff --name-only main...HEAD
bun run test:unit
bun run test:e2e
bun run docker:test
```

## Формат финального ответа пользователю
1. Что изменилось (коротко).
2. Какие тесты выбраны и почему.
3. Какие результаты.
4. Вердикт.
5. Следующий шаг.

Пример:
```text
Изменения: API + migrations
Запущены: unit, e2e, docker:test
Результат: unit pass, e2e pass, docker:test pass
Вердикт: можно деплоить
Следующий шаг: handoff в fb-deploy-operator
```
