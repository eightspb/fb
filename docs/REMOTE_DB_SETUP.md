# Работа с удалённой базой данных

Руководство по настройке локального фронтенда для работы с продакшн базой данных через SSH туннель.

## 🎯 Зачем это нужно?

**Проблема:** При локальной разработке база данных пустая, нет реальных данных для проверки изменений фронтенда.

**Решение:** Подключить локальный Next.js к продакшн базе данных через безопасный SSH туннель.

**Преимущества:**
- ✅ Работаете с реальными данными из продакшн базы
- ✅ Не нужно деплоить каждое изменение для проверки
- ✅ База данных остаётся защищённой (порт не открыт в интернет)
- ✅ Продакшн приложение работает без изменений
- ✅ Можно быстро переключаться между локальной и удалённой базой

```
┌─────────────────────┐
│  Локальный Next.js  │
│   localhost:3000    │
└──────────┬──────────┘
           │ DATABASE_URL=localhost:54321
           ↓
┌─────────────────────┐
│    SSH Tunnel       │
│   localhost:54321   │
└──────────┬──────────┘
           │ SSH (зашифровано)
           ↓
┌─────────────────────┐
│  Сервер (155.x.x.x) │
│                     │
│  ┌───────────────┐  │
│  │ Docker: app   │  │
│  │ (работает как │  │
│  │  обычно)      │  │
│  └───────┬───────┘  │
│          │          │
│  ┌───────▼───────┐  │
│  │ Docker:       │  │
│  │ PostgreSQL    │◄─┼─ SSH туннель подключается сюда
│  │ (fb-net-db)   │  │

└─────────────────────┘
```

**Важно:** Продакшн приложение продолжает работать нормально, SSH туннель не влияет на него.

## 📝 Что происходит внутри?

Скрипт `dev-remote.ps1` автоматически:

1. **Проверяет SSH подключение** к серверу
2. **Проверяет, что PostgreSQL контейнер запущен** на сервере
3. **Создаёт SSH туннель**: `localhost:54321` → `server:5432` (PostgreSQL в Docker)
4. **Запускает Next.js dev сервер** с подключением к `localhost:54321`
5. **При Ctrl+C автоматически закрывает** туннель и dev сервер

## 🛡️ Безопасность

### Почему это безопасно?

✅ **База данных защищена:**
- Порт PostgreSQL (5432) **не открыт** в интернет
- Доступ только через SSH (требуется SSH ключ)
- Используется существующая SSH аутентификация

✅ **Нет изменений на сервере:**
- Не нужно открывать порты в firewall
- Не нужно менять настройки PostgreSQL
- Не нужно менять Docker конфигурацию

✅ **Секреты не храним в репозитории:**
- Пароль БД не должен появляться в `.md`, `package.json`, коммитах и PR
- Храним только локально: в `.env.local` (в gitignore) или в менеджере секретов

✅ **Продакшн не затронут:**
- Приложение `fb-net-site` работает как обычно
- База данных `fb-net-db` работает как обычно
- SSH туннель - это просто дополнительное подключение

### Что изменилось в проекте?

**Локальные изменения (только на вашем компьютере):**
- `.env.local` - добавлен `DATABASE_URL` для туннеля (порт 54321)
- `package.json` - добавлены npm скрипты (`dev:remote`, `tunnel:start`, `tunnel:check`)
- `scripts/dev-remote.ps1` - новый PowerShell скрипт для автоматизации

**На сервере:** 0 изменений ✅

## 🐛 Troubleshooting

### Ошибка: "Не удалось подключиться к серверу"

**Причина:** SSH подключение не работает.

**Решение:**
```powershell
# Проверьте SSH подключение
ssh root@155.212.217.60

# Если не работает - настройте SSH ключи
# Или проверьте ~/.ssh/config
```

---

### Ошибка: "PostgreSQL контейнер не запущен"

**Причина:** Контейнер `fb-net-db` не запущен на сервере.

**Решение:**
```powershell
# Запустите контейнер на сервере
ssh root@155.212.217.60 "cd /opt/fb-net && docker compose -f docker-compose.production.yml up -d postgres"

# Проверьте статус
ssh root@155.212.217.60 "docker ps | grep fb-net-db"
```

---

### Ошибка: "Порт 54321 уже используется"

**Причина:** SSH туннель уже запущен или порт занят другим процессом.

**Решение 1 - Закрыть существующий туннель:**
```powershell
# Найти процесс на порту 54321
Get-NetTCPConnection -LocalPort 54321 | Select-Object OwningProcess

# Закрыть процесс (замените PID)
Stop-Process -Id <PID> -Force
```

**Решение 2 - Использовать другой порт:**

Измените в файлах:
- `.env.local`: `DATABASE_URL=...@localhost:54322/...` (вместо 54321)
- `package.json`: `tunnel:start` с `-L 54322:localhost:5432`
- `scripts/dev-remote.ps1`: параметр `-LocalPort 54322`

---

### Ошибка: "Cannot connect to database"

**Причина:** Туннель создан, но подключение к базе не работает.

**Решение:**
```powershell
# 1. Проверьте, что туннель работает
netstat -an | findstr 54321

# 2. Проверьте пароль в .env.local
# Должен быть URL-encoded и подставлен через локальную переменную:
# REMOTE_DB_PASSWORD_URLENCODED=<ваш URL-encoded пароль>

# 3. Проверьте логи PostgreSQL на сервере
ssh root@155.212.217.60 "docker logs fb-net-db --tail=50"

# 4. Проверьте, что база принимает подключения
ssh root@155.212.217.60 "docker exec fb-net-db psql -U postgres -d postgres -c 'SELECT 1;'"
```

---

### База данных "read-only" или изменения не сохраняются

**Это нормально!** Вы работаете с **продакшн базой данных**.

**Важно:**
- ⚠️ Все изменения в базе **реальные** и влияют на продакшн
- ⚠️ Не удаляйте и не изменяйте данные без необходимости
- ✅ Для тестирования используйте локальную базу (`bun run docker:up && bun run dev`)

---

### Туннель закрывается сам по себе

**Причина:** SSH сессия разрывается из-за таймаута.

**Решение - Настроить SSH keep-alive:**

Добавьте в `~/.ssh/config`:
```
Host 155.212.217.60
    ServerAliveInterval 60
    ServerAliveCountMax 3
```

Или запустите туннель с параметрами:
```powershell
ssh -N -L 54321:localhost:5432 -o ServerAliveInterval=60 root@155.212.217.60
```

## 📚 Дополнительная информация

### Почему порт 54321, а не 5432?

- `5432` - стандартный порт PostgreSQL (может быть занят локальным PostgreSQL)
- `54322` - используется локальным Docker PostgreSQL (`docker-compose.yml`)
- `54321` - используется для SSH туннеля к продакшн базе

Это позволяет одновременно иметь:
- Локальную базу на `54322`
- Удалённую базу на `54321` (через туннель)

### Как работает SSH туннель?

```
ssh -N -L 54321:<CONTAINER_IP>:5432 root@155.212.217.60
```

- `-N` - не выполнять команды, только туннель
- `-L 54321:<CONTAINER_IP>:5432` - перенаправить локальный порт 54321 на IP контейнера PostgreSQL в Docker сети
- `<CONTAINER_IP>` - IP адрес Docker контейнера (например, 172.18.0.3), получается автоматически скриптом
- **Важно:** PostgreSQL контейнер не публикует порты наружу, поэтому нужно подключаться к его IP в Docker сети

### Альтернативы (не рекомендуется)

**❌ Прямое подключение:**
```env
DATABASE_URL=postgresql://postgres:${DB_PASSWORD_URLENCODED}@155.212.217.60:5432/postgres
```
Требует открытия порта 5432 в интернет - **небезопасно!**

**❌ Proxy API:**
Создать отдельный API сервер на продакшене для проксирования запросов - **избыточно сложно**.

**✅ SSH туннель (используется):**
Безопасно, просто, не требует изменений на сервере.

1. **Для разработки фронтенда** - используйте удалённую базу (`.\scripts\dev-remote.ps1`)

4. **Закрывайте туннель** после работы (Ctrl+C)
5. **Используйте осмысленные коммиты** при изменении схемы базы
